2. 핵심 기술 구조 (최종 채택 방식)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[왜 빠른가?]
  기존: Playwright로 크롬 브라우저를 직접 조작 → 각 필드 클릭 → 입력 → 저장
        → 건당 수십 초, 사람이 보는 것과 동일한 속도

  최종: 로젠 내부 REST API를 직접 호출 (HTTP POST)
        → UI 없이 데이터만 전송 → 건당 1초 이하
        → 브라우저 자동화가 아닌 API 자동화

[로젠 등록 4단계 API 흐름]
  ① POST /lrm01b0050/saveResv      → 운송장 예약
  ② POST /lrm01b0050/getResvList   → 예약 확인 (seq 번호 추출)
  ③ POST /lrm01b0050/getSlipNoSave → 운송장 번호 채번
  ④ POST /lrm01b0050/updateSlipNo  → 번호 확정 (이 단계에서 실제 등록 완료)

  Base URL: https://logis.ilogen.com
  인증 헤더: x-token: {JWT}, x-pageId: lrm01f0050

[JWT 인증 구조]
  로젠 사이트 로그인 후 document.body.id 에 JWT가 저장됨.
  (로젠 자체 JS인 BASE_PC_V2.js 가 세팅함)
  이 JWT를 가져다 API 호출 시 헤더에 실어 보내면 인증 완료.

3. 시행착오 1 — Playwright 브라우저 직접 조작의 한계
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[시도한 방법]
  Playwright로 logis.ilogen.com 브라우저를 열고,
  각 입력 필드를 찾아서 값을 입력하고, 저장 버튼을 클릭하는 방식.

[문제점]
  - 속도 너무 느림 (건당 20~30초)
  - 사이트 구조 변경 시 바로 깨짐 (셀렉터 변경)
  - iframe, 팝업 등 예외처리 복잡
  - 대량 처리 시 브라우저 리소스 과부하

[해결책]
  브라우저 조작 방식을 완전히 버리고,
  개발자도구(F12) Network 탭으로 로젠 사이트가 실제로 호출하는
  API 엔드포인트를 직접 분석 → 그 API를 파이썬으로 직접 호출.

[교훈]
  웹 자동화 시 항상 먼저 "이 사이트가 내부적으로 API를 쓰는가?" 확인.
  REST API가 있으면 브라우저 조작보다 API 직접 호출이 압도적으로 유리.
  (속도, 안정성, 유지보수 모두)

4. 시행착오 2 — Claude in Chrome computer 도구 타임아웃
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[문제]
  Claude MCP의 computer 도구(screenshot, click)가
  logis.ilogen.com 에서 계속 타임아웃 발생.
  로그인 버튼을 찾아서 클릭하는 것 자체가 안 됨.

[시도한 것들]
  - getBoundingClientRect()로 좌표 계산 후 click → 타임아웃
  - 여러 셀렉터로 버튼 찾기 → 타임아웃
  - screenshot 도구로 화면 확인 → 타임아웃

[해결책]
  javascript_tool 로 로젠 사이트 자체 JS 함수를 직접 호출:
    basicLogin()  → 로젠이 자체적으로 만든 로그인 함수

  브라우저 DOM 클릭 대신 JS 함수 직접 실행으로 우회 성공.

[추가 문제: 비밀번호 자동완성]
  Chrome 보안 정책상 JS로 password 필드 값을 읽으면 빈 문자열 반환.
  (실제로는 자동완성 되어있어도 JS에서 못 읽음)

  해결: pw.value.length 로 길이만 확인 (0이면 미입력, 아니면 입력됨)
        → 길이가 0이 아니면 basicLogin() 호출.

[추가 문제: swal 팝업 "비밀번호를 입력하세요" 모달]
  비밀번호 없이 로그인 시도 시 SweetAlert 팝업이 떴다가
  그 팝업이 모든 동작을 막음.

  해결: document.querySelector('.swal-button').click() 으로 팝업 닫기

[교훈]
  MCP computer 도구(screenshot/click)는 일부 사이트에서 타임아웃.
  javascript_tool 이 훨씬 빠르고 안정적.
  가능하면 사이트 자체의 JS 함수를 직접 호출하는 방식 우선 시도.

5. 시행착오 3 — JWT 토큰 관리
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[초기 방식]
  서버(run_server.py) 메모리에만 JWT 저장.
  → 서버 재시작하면 JWT 날아감 → "로젠 로그인 필요" 오류 반복

[개선 1: 디스크 영속화]
  서버 수신 시 lc_config.json 에도 동시 저장.
  서버 시작 시 lc_config.json 에서 자동 복원.
  → 재시작해도 JWT 유지.

[개선 2: 6시간 자동 갱신]
  로젠 JWT는 일정 시간 후 만료됨.
  threading.Timer 로 6시간마다 logen_get_token.py --headless 자동 실행.
  → 사용자가 신경 쓸 필요 없이 항상 유효한 JWT 유지.

[Playwright 세션 영속화]
  .playwright_logen/ 폴더에 브라우저 세션(쿠키) 저장.
  → 최초 1회 수동 로그인 후 이후는 headless로 자동 로그인 가능.

[교훈]
  JWT/토큰 관리는 항상 3단계로 설계:
    1) 메모리 (빠른 접근)
    2) 디스크 (재시작 후 복원)
    3) 자동 갱신 스케줄러 (만료 대비)

6. 시행착오 4 — 데이터 필드 매핑 오류
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[증상]
  로젠 등록은 빠르게 되는데 전화번호, 주소, 배송메시지가 빠져있음.
  로젠 사이트에서 확인하면 수하인 정보가 텅텅 비어있음.

[원인 1: 필드명 불일치]
  카페24 API 응답 필드명과 코드에서 사용한 변수명이 달랐음.

  잘못된 코드:
    phone:   item.receiverPhone    (존재하지 않는 필드)
    address: item.receiverAddress  (존재하지 않는 필드)

  올바른 코드:
    phone:   item.receiverTel      (카페24 실제 필드명)
    address: item.receiverAddr     (카페24 실제 필드명)

[원인 2: 주소 분리 미처리]
  카페24는 주소를 address1(도로명주소)과 address2(상세주소)로 분리.
  로젠도 주소와 상세주소를 별도 필드로 받음.
  → address1 + address2 를 하나로 합치거나 address2 를 누락시키면 안 됨.

  올바른 매핑:
    receiver.address1 → rcvCustAddr    (로젠 주소 필드)
    receiver.address2 → rcvCustAddrDTL (로젠 상세주소 필드)

[원인 3: 배송메시지 필드 누락]
  normalizeOrder() 함수에서 shipping_message 를 아예 안 읽어왔음.

  추가:
    shippingMessage: receiver.shipping_message || ''

[교훈]
  외부 API 연동 시 반드시 실제 응답 JSON을 console.log로 출력해서
  필드명을 직접 눈으로 확인할 것.
  "아마 이런 이름일 거야"로 추측하면 반드시 틀림.

7. 시행착오 5 — 설정 저장 (localStorage → 서버 파일)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[초기 방식]
  카페24 API 키, mallId 등 설정을 브라우저 localStorage에만 저장.

[문제]
  바탕화면 바로가기로 실행하면 브라우저 프로필이 달라질 수 있음.
  또는 다른 브라우저에서 열면 설정이 없음.

[해결]

  localStorage는 브라우저 종속적이라 한계 있음.

8. 시행착오 6 — Python 서버 프로세스 중복 문제
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

[문제]
  개발 중 서버를 여러 번 재시작하면서 포트 8080을 점유한
  구 Python 프로세스가 백그라운드에 누적됨.
  → 새 코드로 서버를 시작해도 구 프로세스가 요청을 먼저 받아버림.
  → 코드는 수정했는데 동작이 안 바뀌는 것처럼 보여 혼란.
  → /api/config 가 "Not Found" 반환 (구 버전 서버가 응답)

[해결]
  PowerShell로 모든 Python 프로세스 강제 종료:
    Get-Process python | Stop-Process -Force

  이후 명시적 작업폴더 지정해서 재시작:
    Start-Process python -ArgumentList "-X utf8 run_server.py" `
      -WorkingDirectory "C:\...\kuaai-main"

  bat 파일에도 시작 전 기존 프로세스 종료 로직 추가:
    for /f "tokens=5" %%a in ('netstat -ano | findstr ":8080 " | findstr "LISTENING"') do
      taskkill /PID %%a /F

[교훈]
  로컬 개발 서버는 재시작 시 반드시 기존 프로세스 종료 확인.
  "코드 수정했는데 왜 안 바뀌지?" → 거의 항상 구 프로세스 문제.
  bat 파일에 포트 점유 프로세스 자동 종료 로직 기본으로 넣을 것.

      ↕ fetch
  로컬 Python 서버(run_server.py, 포트 8080)  ← 프록시 + 설정관리
      ↕ HTTP POST
  카페24 API (주문 조회 / 송장 업로드)
      ↕ HTTP POST
  로젠 API (운송장 등록 4단계)

[JWT 흐름]
  Playwright(logen_get_token.py) → 로젠 로그인 → document.body.id 에서 JWT 추출
  → POST /store-logen-token → 메모리 + lc_config.json 저장
  → 6시간마다 자동 갱신 (threading.Timer)

[왜 로컬 Python 서버가 필요한가?]
  1) CORS: 브라우저에서 외부 API 직접 호출 불가 → 서버가 프록시
  2) JWT 보관: 브라우저 메모리보다 서버가 안전하고 영속적
  3) 설정 관리: lc_config.json 읽기/쓰기
  4) 자동화 스크립트 실행: subprocess로 Playwright 스크립트 실행
